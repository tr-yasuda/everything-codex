# Code Review

## Inspect The Change First

レビュー前に次を確認する。

- 変更の目的、対象ファイル、差分の範囲
- 既存の規約、テスト構成、公開 API の有無
- レビュー対象が新規実装か、既存コードの修正か

目的を把握する前に指摘を始めない。

## Start With Findings

返答では、要約より先に指摘を並べる。

- 指摘は重大度順に並べる
- 各指摘に、影響・根拠・修正方針を含める
- 可能な限りファイルと行を示す
- 問題が見つからなければ、その旨を明示する

## Check The Review Axes

次の 9 観点を漏れなく確認する。

- 正確性: ロジック、境界値、並行性、エラー処理に破綻がない
- セキュリティ: 外部入力、認証・認可、機密情報の扱いが安全である
- パフォーマンス: 無駄な I/O、N+1、再計算、大量データ処理に無理がない
- 後方互換性: 公開 API、保存形式、設定変更が既存利用者を壊さない
- 可観測性: 障害解析に必要なログ、メトリクス、エラー文脈が不足しない
- 運用性: 設定、デプロイ、ロールバック、障害時対応が現実的である
- テスト容易性: テストしやすい分離になっており、正常系・異常系・境界値を確認しやすい
- 可読性: 意図が名前、責務、構造から素直に読める
- 一貫性: 既存の規約、設計、命名、データの扱いと揃っている

破壊的変更や運用前提がある場合は、その前提を明示する。

## Keep Evidence Thresholds Clear

diff や周辺文脈から根拠が取れる範囲だけを断定する。

- セキュリティ、パフォーマンス、運用リスクは根拠がある場合だけ強く指摘する
- 可観測性やテスト容易性は、障害解析や検証を阻害する理由まで添える
- 可読性や一貫性は、実害や保守コストに繋がる場合を優先する

## Treat Ambiguity As A Stop Signal

次のケースは曖昧扱いにする。

- 変更の目的や背景が読み取れない
- 既存規約が複数あり、どれを基準にするか決められない
- 互換性や破壊的変更の評価に必要な周辺文脈が足りない

曖昧な場合は、不足情報と確認事項を返して止まる。

## Review Before Finishing

終了前に次を確認する。

- 指摘を返す場合は、要約より先に findings を並べている
- 各指摘に、重大度・根拠・影響・修正方針がある
- 問題なしの観点は、必要な場合だけ短く補足している
- 確認できていない領域は、未確認として分けている
