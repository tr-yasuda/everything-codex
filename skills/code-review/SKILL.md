# Code Review

## Inspect The Change First

レビュー前に次を確認する。

- 変更の目的、対象ファイル、差分の範囲
- 既存の規約、テスト構成、公開 API の有無
- レビュー対象が新規実装か、既存コードの修正か

目的を把握する前に指摘を始めない。

## Start With Findings

返答では、要約より先に指摘を並べる。

- 指摘は重大度順に並べる
- 各指摘に、影響・根拠・修正方針を含める
- 可能な限りファイルと行を示す
- 問題が見つからなければ、その旨を明示する

## Check Correctness And Regression Risk

コードが意図どおりに動くかと、既存動作への影響を確認する。

- ロジックに誤りや抜け漏れがない
- 境界値、null、空配列などの edge case を扱っている
- 非同期処理の順序や競合状態に問題がない
- エラーが適切に捕捉・伝播されている
- 呼び出し元や周辺フローに回帰が出ない
- 名前や構造が、読み違いによるバグを招かない

## Check Test Coverage And Quality

テストが妥当かを確認する。

- 正常系・異常系・境界値がカバーされている
- テストが実装の詳細ではなく振る舞いを検証している
- テストが独立して実行でき、順序依存がない
- テストの意図が名前や構造から読み取れる

## Check Compatibility And Migration Risk

互換性や移行リスクを確認する。

- 公開 API のシグネチャや戻り値の型が変わっていない
- 既存の呼び出し元が影響を受けない
- データベーススキーマや保存形式の後方互換性がある
- 設定ファイルや環境変数の変更が既存環境を壊さない

破壊的変更の可能性がある場合は、その前提を明示する。

## Check Security And Performance When Evidence Exists

diff や周辺文脈から根拠が取れる範囲だけを扱う。

- 外部入力を危険な形で使っていない（SQL、コマンド、パスなど）
- 認証・認可や機密情報の扱いに問題がない
- ループ内で不必要な重い処理や API 呼び出しをしていない
- N+1 クエリや不要な再レンダリングが発生していない
- キャッシュや遅延評価が適切に使われている
- 大量データを一括処理する箇所にリスクがない
- 脆弱性や性能問題は、ツール出力や diff の根拠がある場合だけ断定する

## Treat Ambiguity As A Stop Signal

次のケースは曖昧扱いにする。

- 変更の目的や背景が読み取れない
- 既存規約が複数あり、どれを基準にするか決められない
- 互換性や破壊的変更の評価に必要な周辺文脈が足りない

曖昧な場合は、不足情報と確認事項を返して止まる。

## Review Before Finishing

終了前に次を確認する。

- 指摘を返す場合は、要約より先に findings を並べている
- 各指摘に、重大度・根拠・影響・修正方針がある
- 問題なしの観点は、必要な場合だけ短く補足している
- 確認できていない領域は、未確認として分けている
