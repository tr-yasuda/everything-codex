# Test-Driven Development

## Inspect The Request And Test Path First

最初に次を確認する。

- 何の振る舞いを変える依頼か
- 既存のどのテストが対象になりそうか
- 最小で回せるテストコマンドは何か（特定できない場合は次のセクションへ）
- 変更がコード対象か、docs や設定だけか

既存テストがあるなら、まず現在の状態で実行して基準を把握する。
対象を絞れないまま広いテストスイートを回す判断をしない。

## Treat Missing Testability As A Stop Signal

次のケースは、そのまま TDD を進めない。

- 安全に実行できる自動テスト手段が見つからない
- docs や文言だけの変更で、コードの振る舞いを対象にしていない
- 外部依存が強く、ローカルで失敗と成功を判定できない
- どの振る舞いを先に固定すべきか曖昧

この場合は、足りない前提と必要な確認事項を返す。
TDD を装って実装だけ先に進めない。

## Pair With Coding Standards On Implementation Tasks

実装タスクでは、`$coding-standards` を必ず併用する。

- `$tdd` が Red-Green-Refactor の進行順序を主導する
- `$coding-standards` は命名、型、エラー処理、変更粒度の
  制約を与える役割として使う
- Plan では failing test、最小テストコマンド、適用する規約を
  まとめて示す
- 規約確認を理由として、Red を置かず先に実装へ進まない

## Write One Failing Test

最初に、変更したい振る舞いを表す最小のテストを 1 つ追加または更新する。

- 1 回で多くをカバーしようとしない
- 公開された振る舞いをテストする
- テスト名は「何の振る舞いを、どんな条件で、どう期待するか」を表す
- 対象コードが未実装で失敗するなら、その失敗を Red として扱う
- テストを実行し、期待どおりに失敗することを確認する

失敗理由がテストの書き間違いなら、先にテストを直してから実装に進む。

## Write Minimal Code To Reach Green

直前に追加した Red を Green にする最小限の実装だけを書く。

- ついでの最適化、設計変更、横展開を混ぜない
- 必要なら一時的に重複や素朴な分岐を許容する
- 実装後は対象テストをすぐ再実行する
- まだ失敗するなら、原因を絞って再度最小実装を足す

## Refactor Only Under Green

対象テストが通ったあとでだけ、構造を改善する。

- 重複を除去する
- 命名を改善する
- 小さな責務分離や抽出をする
- リファクタリングの前後で対象テストを再実行する

回帰が出たら、先に Green へ戻す。
失敗したまま次の改善に進まない。

## Keep The Change Scope Small

1 サイクルでは 1 つの振る舞いだけを進める。

- 新しい要件は次の Red として分ける
- 無関係なファイルや広いリファクタリングを抱き合わせにしない
- テスト追加、最小実装、改善の境界を答えでも明確にする

Git 操作やコミットは明示依頼がある場合だけ行う。
フェーズの区切りは説明で示し、勝手に commit しない。

## Review Before Finishing

終了前に次を確認する。

- 最後に追加した振る舞いを表すテストが通っている
- 関連する既存テストも必要な範囲で通っている
- 追加した実装が最小限に保たれている
- カバレッジの著しい低下がないか確認している（任意）
- 残るテストギャップや未対応範囲を共有している

返答では、どのテストを Red にし、どう Green にし、
何を Refactor したかを簡潔に示す。
