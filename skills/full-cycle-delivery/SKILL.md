# Full-Cycle Delivery

## Start From The Requested Outcome

最初に、依頼が最終的に何を達成したいのかを一文で固定する。

- 依頼の表現ではなく、達成したい結果を書く
- 成果物がコード、設定、文書、調査結果のどれかを切り分ける
- 完了条件が曖昧なら、作業前に明示する

意図が読み切れない場合は、推測で広げず `何を終わりとするか` を優先して
確認する。

## Triage The Task Before Acting

着手前に、依頼の種類を先に分類する。

- 仕様追加か
- バグ修正か
- リファクタリングか
- レビューや調査か
- ドキュメント整備か

分類によって、使う skill と検証方法を切り替える。
分類を曖昧にしたまま、実装や提案を始めない。

## Pull In The Right Supporting Skills

この skill は単独で抱え込まず、必要な skill を明示的に束ねる。

- 依頼の意図や制約が曖昧なら `$task-intake` を先に使う
- リポジトリ構成やコマンドが不明なら `$repo-discovery` を使う
- 振る舞い変更を伴う実装は `$tdd` を主導にする
- 振る舞いを変えない構造改善は `$refactoring` を併用する
- 実装規約の統一が必要なら `$coding-standards` と
  言語別 skill を併用する
- 言語別 skill の例は `$typescript-practice`、`$rust-practice`、
  `$terraform-practice` である
- 仕上げで差分の抜け漏れが不安なら `$code-review` を使う
- ブランチ、commit、PR まで進めるなら関連 skill を最後につなぐ

どの skill を使うかは、目的と変更の種類に対して理由付きで選ぶ。
使わない skill も、なぜ不要かを説明できる状態にする。

## Design The Smallest Valuable Slice

作業は、最小の価値単位に分割する。

1. 再現を切り出す
2. 修正を切り出す
3. 仕上げを切り出す

- 一度に複数の問題を直そうとしない
- 共有モジュールや公開面の変更は、先に波及範囲を洗う
- ついでの整理は、完了条件に必要なものだけに絞る

大きな依頼でも、最初の 1 手は `今すぐ検証できる小さな差分` に落とす。

## Verify In Rising Cost Order

検証は、安く狭いものから順に積み上げる。

- まず再現手順、対象テスト、lint、typecheck などの最小確認を決める
- 次に関連する既存確認を広げる
- 最後に必要な統合確認や手動確認をする
- 実行していない確認は、実行済みとして扱わない

バグ修正では、修正前に失敗を観測し、修正後に同じ経路で解消を確かめる。
文書変更では、参照先パスとコマンドの実在を必ず確認する。

## Keep Reasoning Visible And Tight

返答や進捗共有は、判断材料が追える粒度に保つ。

- 何を事実として確認したか
- 何を仮定として置いたか
- なぜその手順と検証を選んだか
- どこまで完了し、何が未了か

長い説明より、次の判断に必要な情報を優先する。
曖昧な達成宣言や、検証なしの自信は避ける。

## Escalate High-Impact Unknowns Early

次の不明点は、抱えたまま進めない。

- 完了条件が複数解釈できる
- 破壊的変更の可能性がある
- 依存追加や大きな方針変更が必要
- ローカルでは真偽を判定できない

質問は少数に絞り、回答によって計画がどう変わるかも添える。

## Review Before Finishing

完了前に次を確認する。

- 目的、制約、影響範囲、完了条件が答えに反映されている
- 変更は最小差分に保たれている
- 必要な検証を済ませ、結果を説明できる
- 未解決事項、残リスク、次の一手が明示されている
- 必要な supporting skill を使い分けられている

返答では、`何を達成したか`、`どう検証したか`、
`何が残っているか` を短く具体的に示す。
