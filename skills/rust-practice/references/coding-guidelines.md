# コーディングガイドライン（Rust）

このドキュメントは、Rust 変更時の必須規約を定義します。
迷った場合はこのドキュメントを優先します。

## 基本原則

- 可読性・保守性・後方互換性を最優先する。
- まず安全な実装を選ぶ。`unsafe` は最小範囲で閉じ込める。
- 差分は最小化し、不要な設計変更を混ぜない。

## バージョンと互換性

- `Cargo.toml` に `edition` と `rust-version` を明示する。
- 新規 crate は原則 `edition = "2024"` を使う。
- MSRV を上げる変更は、変更理由と影響範囲を明示する。

## API 設計

- 公開範囲は最小化する。`pub` は必要最小限にとどめる。
- 公開 API は意図が伝わる命名を優先する。
- 公開 API、公開型、シリアライズ形式はコントラクト層として先に固定する。
- コントラクト層を実装詳細や内部最適化へ引きずられない形で保つ。
- 公開 API がエラー・`panic`・`unsafe` を持つ場合は、次を明記する。
  - `# Errors`
  - `# Panics`
  - `# Safety`

## 状態とロジック

- 共有状態の保持・更新と、判断・変換・検証のロジックを分離する。
- I/O、同期制御、永続化の都合を、純粋ロジックへ漏らしすぎない。
- 実装層は差し替え可能な詳細として保ち、
  生成物や補助実装を手修正の唯一の正にしない。

## エラー処理

- ライブラリ層では、呼び出し側が扱いやすいエラー型を返す。
- アプリケーション境界では、文脈を付与してエラーを伝播する。
- 本番経路での `unwrap` / `expect` は禁止とする。
- `unwrap` / `expect` はテストコードまたは不変条件が証明済みの箇所に限定し、理由コメントを付ける。

## 所有権とデータコピー

- 不要な `clone` を避ける。まず借用で表現できるか確認する。
- ただし、可読性を著しく下げる借用最適化は行わない。
- ライフタイム注釈は必要なときだけ追加する。

## `unsafe` 運用

- `unsafe` は安全抽象の内部へ隔離する。
- `unsafe` ブロック直前に安全性の前提条件をコメントで記述する。
- `unsafe fn` 内でも必要箇所を個別の `unsafe {}` で囲む。
- FFI 境界・生ポインタ操作・共有可変状態は特に厳密にレビューする。

## 非同期・並行処理

- `async` 文脈でブロッキング呼び出しを避ける。
- `Send` / `Sync` 制約を壊す設計は、理由と代替案を明示する。
- 共有状態はロック粒度と競合可能性を説明できる設計にする。

## テスト

- 変更点に近いテストを優先して追加・更新する。
- 正常系だけでなく、失敗系と境界条件を含める。
- 公開契約と失敗経路の保護範囲を狭めない。
- `panic` を期待するテストは、理由が明確なケースに限定する。

## Lint / Format

- フォーマットは `cargo fmt` を正とする。
- `clippy` は原則 `-D warnings` で運用する。
- lint の `allow` は最小範囲に限定し、理由コメントを必須にする。
- `clippy::pedantic` / `clippy::restriction` の一括 `deny` は避ける。必要項目のみ選抜する。
