# Refactoring

## Inspect The Target First

最初に次を確認する。

- 対象が巨大関数・fat component・util の肥大化・型の崩れのどれか
- 既存のテストや型チェックで振る舞いを保護できるか
- 変更前に通る最小の test / lint / typecheck の単位
- 依頼が振る舞い変更を含むか、構造改善だけか

振る舞い変更と構造改善が混在している場合は、先に分離を提案する。

## Require A Safety Net First

安全網が弱い場合は、先に補強する。

- 既存の test / typecheck が使えるなら、最小単位で先に確認する
- 安全網が不足しているなら、まず `$tdd` で最小の保護を追加する
- 手動メモや目視確認は補助として使っても、安全網の代替にしない
- 安全に切り出せる単位や保護手段を特定できない場合は止まる

## Separate Responsibilities

責務分離では次を行う。

- 1 つの関数・コンポーネントが担っている役割を列挙する
- データ取得・変換・表示・副作用など、異なる関心を分ける
- 分離後の単位が単一の責務を持つことを確認する
- 分離は依頼に必要な範囲へ留め、無関係な箇所まで広げない

## Split Large Functions

巨大関数の分割では次を行う。

- 関数内のまとまりを意味のある処理単位で識別する
- 抽出した関数には、処理の意図を表す名前を付ける
- 引数と戻り値を明示し、副作用を局所化する
- 分割後に元の関数の振る舞いが変わっていないことを確認する

## Organize Types

型の整理では次を行う。

- `any` や曖昧な型を `unknown` と型ガードで置き換える
- 重複した型定義を共通の型へ統合する
- ドメインの意味を反映した型名に見直す
- 公開 API と内部型の境界を明確にする

## Revise Naming

命名の見直しでは次を行う。

- 実装都合ではなく、ドメインの意味で名前を付ける
- `data`、`info`、`temp`、`util` のような曖昧名を具体化する
- 周辺コードの命名規約に合わせる
- rename の影響範囲を依頼で必要な最小限に留める

## Remove Duplication

重複排除では次を行う。

- 同じロジックが複数箇所にある場合、共通化の単位を特定する
- 共通化が可読性を下げる場合は、重複を残す判断をする
- 抽出した共通処理には、再利用の意図を表す名前を付ける
- 共通化後に既存の呼び出し元の振る舞いが変わっていないことを確認する

## Keep One Improvement Per Pass

1 回の変更では 1 つの改善だけを進める。

- 責務分離・関数分割・型整理・命名・重複排除を同時に混ぜない
- 無関係なファイルへ広げない
- 各ステップの前後で振る舞いが維持されていることを確認する

## Confirm Behavior Is Preserved

振る舞い維持の確認では次を行う。

- リファクタリング前後で test / lint / typecheck を実行する
- 回帰が出た場合は、次の整理へ進まず先に戻す
- 意図的な振る舞い変更は、別タスクか別差分として切り出す
- 実行できなかった確認項目は、理由付きで残す

## Pair With Other Skills

次の場合は専用 skill と連携する。

- 安全網を追加しながら進める場合: `$tdd`
- 命名・型・変更粒度の規約を揃える場合: `$coding-standards`

## Review Before Finishing

返答前に次を確認する。

- 振る舞いが変わっていない
- 変更単位が安全網で保護されている
- 分割・抽出した単位が単一の責務を持っている
- 型が明示的で、`any` が残っていない
- 命名がドメインの意味を反映している
- 重複が意図的に残されている場合は、その理由を示している
- 実行した test / lint / typecheck を明示している
